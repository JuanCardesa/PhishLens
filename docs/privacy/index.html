Privacy Policy

Last updated: June 2026 — PhishLens v0.3.0

PhishLens is designed to minimise data collection. This policy describes exactly what the extension processes and what it never touches.

What PhishLens processes

  • The URL of the current page for popup analysis and local badge scoring. Page-load badge scoring stays inside the extension; backend enrichment is requested from the popup.
  • Derived URL features: length, dot and hyphen counts, IP domain detection, subdomain count, presence of suspicious keywords in the hostname and path, punycode detection, and domain entropy.
  • Non-sensitive page-structure counts: number of forms, presence of a password-input element, whether any form submits to an external domain, number of iframes, ratio of external links, and presence of hidden inputs.
  • Local brand-mismatch booleans derived from limited page metadata: document title, og:site_name, first heading, and favicon URL. Raw page text and favicon bytes are not sent to the backend, logged, stored, or included in reports.
  • Optional TLS certificate metadata (issuer, expiry) fetched from the domain when TLS analysis is enabled on the companion backend.
  • Optional PhishTank lookup result (in-database, verified, valid flags) when threat intelligence is enabled on the companion backend.
  • Optional user feedback: if you click "Mark as safe" or "Mark as phishing", the extension sends the current URL, the displayed label, and the expected label to the backend. The backend stores only hostname-level label metadata, note presence, request ID, and timestamp. No page content or note text is included.
  • Aggregate backend diagnostics: request counts, label distribution, cache hit rates, and error counts. These are process-local counters and reset when the backend restarts.

What PhishLens never collects

The following data is never collected, stored, or transmitted by PhishLens:

  • Passwords or password-field values
  • Typed emails or any form field values
  • Full page HTML or page screenshots
  • Cookies or session tokens
  • Browser history
  • Raw private page text or user-generated content
  • Full URLs in backend diagnostics or logs

Local storage

The extension stores two things in chrome.storage:

  • Settings (synced): backend URL, request timeout, and danger-overlay toggle.
  • Analysis cache (local): a short-lived entry (5-minute TTL) keyed by a one-way hash of the current URL, so the popup does not re-analyse the same page on every open.

No analysis cache entry contains passwords, form values, or any sensitive page content.

Backend data handling

The optional companion API (FastAPI, self-hosted) does not durably persist /analyze requests. It can persist /report feedback to a local SQLite database, limited to hostname-level label metadata, note presence, request ID, and timestamp. In-memory caches for TLS and PhishTank results are process-local and cleared on restart.

If you use a self-hosted backend, you control the infrastructure and logging configuration. PhishLens does not send data to any Anthropic or third-party servers.

API keys

PhishTank API keys are backend environment variables only. They are never bundled with the extension or transmitted to the browser.

Permissions

PhishLens requests the minimum permissions needed:

  • activeTab — reads the URL of the current tab when you open the popup. No background scanning.
  • scripting — injects the dismissible danger overlay when a page scores as Dangerous and the overlay setting is enabled.
  • storage — stores extension settings and the short-lived analysis cache.

Changes to this policy

If the data handling described here changes materially, a new version of this policy will be published at this URL and the extension changelog will reference it.

Contact

Questions or concerns about this policy can be raised as a GitHub issue.
